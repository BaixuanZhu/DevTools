# 开发路线图（ROADMAP）

> 本文档是 dev-tools 工具站的**工具补全与演进指导**，规划了从「现有工具深化」到「缺失分类补全」的完整路径，作为接下来开发的选题与排期依据。
>
> - **最后更新**：2026-06-18
> - **基线**：25 个工具 / 13 个分类
> - **维护规则**：每完成一项，在「六、进度追踪」中勾选对应条目，并视情况回填实际工时；新增/调整计划时同步更新本文件，使其始终反映最新方向。
> - **配套文档**：产品行为见 `PRODUCT.md`，视觉与组件规范见 `DESIGN.md`，编码约定见 `CLAUDE.md`。本文档只管「做什么、何时做」。

## 一、背景与现状

### 1.1 现状盘点（基线）

| 分类 | URL 前缀 | 工具数 | 覆盖情况 |
|------|----------|:---:|----------|

### 1.2 核心结论

> **覆盖了「编码/加密」深水区，却漏掉了「前端日常」浅水区。**

当前更像一个偏后端、偏加密的工具站。从「绝大多数开发者」视角看，**正则、颜色、CSS、进制转换、文本处理、图片处理**这些每天都碰的轻量需求反而没覆盖——而这些恰恰最容易做出「打开即用、5 秒出结果」的体验，完全契合产品定位（见 `PRODUCT.md` §Product Purpose）。

此外，`PRODUCT.md` 已规划并在侧边栏展示的 **4 个分类是空的**，对用户是「承诺了却没兑现」，需优先补齐。

## 二、总体策略

| 原则 | 说明 |
|------|------|
| **深化优先于拓面** | 现有 25 个工具中，有明确高价值扩展点的先做；空分类其次 |
| **高频优先于长尾** | 正则 / 颜色 / 进制 / 文本这类周级刚需，排在图片格式转换等长尾前 |
| **复用优先于新建** | 能在现有页面扩展的（如二维码加识别、IPv4 加 IPv6），不新建独立页 |
| **本地计算优先** | 浏览器端能完成的纯计算优先；受 CORS 等限制、体验打折的（如 HTTP 请求）往后排 |

### 工时与排期约定

- 工时单位：**人天（d）**，含 UI + 逻辑 + SEO + 自测；按单人全职估算，仅供排期参考。
- 阶段划分：**P0 → P3 共 4 个阶段**，按优先级串行推进，约 10 周（≈ 43d）。
- 每阶段结束后应有可发布的成果，不跨阶段积压。

### 每个新工具/扩展的通用完成清单

> 下列为「完成一个工具」的标配，下文各工具不再逐条重复，统一对照本清单验收。

- [ ] 在 `src/data/tools.ts` 注册，填写完整的 `name` / `description` / `seoDescription` / `keywords` / `relatedToolIds`
- [ ] 按 `/category/tool-name` 二级路由创建页面，复用 `Layout.astro` / `ToolLayout.astro`
- [ ] 输入格式校验 + 中文内联错误提示（见 `PRODUCT.md` §Error Handling）
- [ ] 「清空」「复制结果」按钮 + 合理默认值
- [ ] 完整 SEO：`<title>` / meta description（120–160 字符）/ keywords
- [ ] 必要时在 `src/data/tool-faqs.ts` 维护问答对（渲染为 `<details>` + FAQ 结构化数据）
- [ ] 跨框架通知用 `CustomEvent('toast')`，不引入全局状态库
- [ ] 遵守安全规则：禁用 `eval/Function(string)`，正则包裹 try-catch（见 `CLAUDE.md` §Security Rules）

## 三、阶段计划

### P0 · 现有工具深化（第 1–2 周，≈ 10d）

**目标**：让已上线工具「功能闭环」，补齐用户点进来后自然期待的配套能力。本阶段 0 新增页面，全部在现有路由内扩展，ROI 最高。

| 现有工具 | 扩展方向 | 功能描述 | 工时 |
|---------|---------|---------|:---:|
| 二维码识别器（新增独立工具）✅ | 新增识别 | brainstorming 后改为**独立工具**（遵循项目细分独立工具模式，非合并进生成器）。已上线 `/media/qr-code-reader`：上传/拖拽/Ctrl+V 粘贴 → jsQR 解码 → URL/邮箱/电话可点击 | 2d ✅ |
| IPv6 子网计算器（新增独立工具）✅ | 新增独立工具 | 新增独立工具 `/network/ipv6-cidr`：BigInt 解析、压缩/展开格式互转、地址类型识别、CIDR 计算、前缀参考表。IPv4 工具不动（brainstorming 后由「合并模式」改为独立工具，对标二维码识别器先例） | 2d ✅ |
| 哈希生成器 | 加 HMAC 模式 | 增加带密钥哈希（HMAC-SHA256/1/512），覆盖 API 签名、Webhook 校验场景，与现有无密钥哈希并列 | 1.5d |
| JSON 系列（format 分类） | 加 JSON→TS | `json-to-ts`：粘贴 JSON 生成 TypeScript interface。复用现有 format 分类的交互范式 | 3d ✅ |
| 日期时间转换器 | 加时间差/倒计时 | 两个时间点之差（天/时/分/秒）、未来时刻倒计时。当前仅做格式互转，缺「计算」能力 | 1.5d |

### P1 · 核心空分类补全（第 3–4 周，≈ 6d）

**目标**：消灭 4 个空分类中**最高频的两个**。正则尤其关键，是 CyberChef / regex101 的主要流量来源。

| 分类 | 新工具 | 功能描述 | 工时 |
|------|-------|---------|:---:|
| 正则工具（空→首开） | 正则表达式 | 左侧输入正则 + 标志位（g/i/m/s/u/y），右侧输入测试文本，**实时高亮所有匹配**、列出每个匹配与其捕获组；内置常用正则速查表（邮箱/手机号/URL/IP 等，一键填入）。与 JSON 格式化同级的开发者刚需 | 3d |
| 颜色工具（空→首开） | 颜色转换器 | HEX/RGB/HSL/HSV 实时互转、屏幕取色器、配色板（互补/类似/三角配色）、**WCAG 对比度检查**（输入前景/背景，给出 AA/AAA 达标判定） | 3d |

### P2 · 前端高频补全（第 5–7 周，≈ 12d）

**目标**：覆盖前端每天都会碰、但目前无处可放的轻量需求；开齐 CSS 分类；补上「进制转换」硬缺口。

| 分类 | 新工具 | 功能描述 | 工时 |
|------|-------|---------|:---:|
| CSS 工具（空→首开） | CSS 单位转换器 | px ⇄ rem ⇄ em ⇄ vw/vh 互转，可设根字号（默认 16px）与设计稿宽度，实时双向 | 2d |
| CSS 工具 | CSS 渐变生成器 | 线性/径向/圆锥渐变可视化调色，所见即所得，一键复制 CSS 代码 | 3d |
| 文本处理（补强） | 进制转换器 | 二/八/十/十六进制互转，支持批量多行、逐位二进制预览。**目前全站无进制工具，是最大遗漏项之一** | 1.5d |
| 文本处理 | 文本处理工具箱 | 大小写转换（含全/半角）、去重、去空行、排序、字数/字节统计、查找替换。把零散文本操作收拢一处 | 3d |
| 文本处理 | 假数据生成器 | Lorem ipsum、随机中英文姓名/邮箱/手机号/地址/UUID，按模板批量生成，方便造测试数据 | 2d |

**目标**：把覆盖面从「编码/加密为主」扩展到「全栈通用」；4 个空分类全部开齐。

| 分类 | 新工具 | 功能描述 | 工时 |

| API 工具（空→首开） | HTTP 请求测试器 | 轻量版 Postman：选方法、编辑 headers/body/query，浏览器 `fetch` 发请求，展示状态码/响应头/响应体与耗时。需提示 CORS 限制 | 4d |
| 格式化 | SQL 格式化器 | SQL 美化/压缩、关键字大写、关键字高亮，支持主流方言关键字 | 2d |
| 格式化 | 代码格式化与压缩 | HTML/CSS/JS 美化与压缩（基于成熟库），服务端模板调试场景 | 3d |

## 五、取舍说明

- **API 工具放最后**：HTTP 请求器受浏览器 CORS 限制，能测范围有限，体验不如原生 Postman，定位为「快速试一下」，优先级低于纯本地计算类。
- **图片/媒体类放后**：浏览器 Canvas 能力够用，但工具偏重，单工具 JS 体积易超 50KB 预算（`PRODUCT.md` §Performance Baselines），需谨慎选库，避免拖累页面性能。
- **未纳入本周期**：
  - 需后端能力的：DNS 查询、端口扫描等
  - 低频重型项：Kubernetes YAML 校验、Nginx 配置生成等
  - 这些可在 M4 后视用户反馈再评估。
